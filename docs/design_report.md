# AI 英语口语陪练比赛项目说明

## 1. 项目背景与意义

英语口语能力是大学生学习、求职和国际交流中的重要能力。传统口语练习通常依赖教师、同伴或线下课堂，练习机会有限，反馈也不够及时。很多学习者在真实场景中遇到的问题并不是完全不会英语，而是不知道如何在特定语境下组织回答，例如面试时如何介绍项目经历，点餐时如何礼貌表达需求，会议中如何提出观点和风险。针对这些问题，本项目设计并实现一个“AI 英语口语陪练”最小可交付版本，通过场景选择、文本对话、音频上传、语法纠错、表达优化、能力评分和课后总结等功能，为用户提供一个可重复练习、可即时反馈、可本地运行的学习工具。

本项目的参赛价值主要体现在三个方面。第一，它将自然语言处理应用到真实学习场景中，体现人工智能技术对语言学习的辅助价值。第二，它强调工程可交付性，在没有付费 API 和复杂基础设施的情况下，仍然通过本地演示模式跑通完整业务流程。第三，它便于比赛展示和项目说明，系统能够保存练习历史并导出 Markdown 总结，使实验过程、测试结果和功能截图都有明确依据。

## 2. 需求分析

系统目标用户是希望练习英语口语表达的学生。用户希望在常见场景中进行对话，例如英语面试、餐厅点餐和商务会议。系统需要提供场景化角色扮演能力，使用户不是孤立地输入句子，而是在一个连续情境中练习表达。功能需求包括场景选择、难度选择、文本输入、音频上传、AI 回复、纠错反馈、五维评分、课后总结、历史保存和总结导出。

考虑到比赛项目的时间限制和成本限制，本项目必须支持两种运行模式。API 模式在检测到 `OPENAI_API_KEY` 时启用，可以调用 OpenAI API 生成更自然的回复、纠错、评分和总结。本地演示模式在没有 API Key 时自动启用，通过预设回复、规则纠错、模拟评分和模板总结完成主要流程。该设计保证项目即使在离线或无付费资源的情况下也能稳定运行。

非功能需求方面，系统应当结构清晰、容易运行、错误处理充分、界面适合截图展示。项目不使用数据库、不做登录注册、不进行复杂前后端分离，历史记录直接保存为 JSON 文件，课后总结导出为 Markdown 文件。

## 3. 系统总体设计

系统采用 Python + Streamlit 的单体应用结构。Streamlit 负责页面渲染、用户交互和会话状态管理。业务逻辑拆分为多个模块：`scenarios.py` 管理场景数据，`feedback.py` 负责 AI 回复与纠错，`evaluator.py` 负责五维评分，`speech_utils.py` 负责音频转写与降级处理，`report_generator.py` 负责课后总结生成，`storage.py` 负责文件保存。

该架构的核心优点是简单可靠。用户启动 `streamlit run app.py` 后即可访问页面，不需要额外启动后端服务。所有模块都通过普通 Python 函数调用，避免了接口联调成本。历史数据存储在 `outputs/` 目录下，方便比赛展示时直接查看 JSON 记录和 Markdown 总结。

系统运行流程如下：用户选择场景和难度，输入英文句子或上传音频；如果音频模型可用，系统尝试转写为英文文本；用户提交后，系统判断当前是否存在 API Key；API 可用时尝试调用模型生成结构化结果，否则进入本地规则流程；随后页面展示 AI 回复、纠错反馈和评分；用户点击生成总结后，系统根据本次对话历史生成 Markdown 总结，并更新最近一次 JSON 练习记录。

## 4. 功能模块设计

场景模块提供三个基础场景：面试、点餐和会议。每个场景包含 AI 角色身份、用户练习目标、常见问题、推荐表达、默认难度和本地回复模板。这样设计可以保证回复内容与场景紧密相关，同时后续可以很方便地扩展新场景。

对话模块是系统主流程。页面采用聊天应用风格，左侧显示用户和 AI 的对话历史，右侧显示最近一次回复、纠错和评分。该布局兼顾连续对话感和比赛展示效果，用户可以一眼看到输入、系统反馈和分数变化。

音频模块支持 wav、mp3 和 m4a 文件。考虑到 Whisper 模型可能没有安装，系统不会强制依赖语音模型，而是通过 `try/except` 检测 `faster-whisper` 或 `whisper`。如果模型不可用，页面提示“当前为演示模式，请使用文本输入或示例转写文本”，保证主流程不受影响。

历史模块使用 JSON 文件保存每次练习。保存字段包括时间、场景、难度、用户输入、AI 回复、纠错反馈、评分结果和课后总结。该方式实现简单，适合比赛项目，也便于教师检查系统输出。

## 5. 对话交互设计

本项目的对话交互强调“真实场景”和“可继续练习”。在面试场景中，AI 扮演英文面试官，会追问项目经历、挑战、结果和岗位匹配度。在点餐场景中，AI 扮演服务员，会询问饮品、菜品偏好、过敏信息和附加需求。在会议场景中，AI 扮演业务同事，会追问观点、风险、时间安排和下一步行动。

难度设置影响系统追问方式。简单模式鼓励用户用一到两个句子回答；中等模式要求补充原因或例子；困难模式要求用户给出结构化回答，包括观点、理由、例子和下一步。这样的设计能够体现练习系统的层次性，也使同一个场景可以适配不同水平的学习者。

## 6. 纠错与反馈机制设计

本地演示模式下，系统使用规则纠错。规则至少覆盖常见错误：`I am agree` 改为 `I agree`，`discuss about` 改为 `discuss`，`He go` 改为 `He goes`，并对 `want to + noun` 给出提示。此外，系统还会检查句首大写、句末标点和句子过短等问题。

反馈内容包括原句、修改后句子、问题说明、更自然表达和推荐替代表达。这样的结构比单纯给出改错结果更有教学价值，因为用户不仅知道哪里错，还能知道为什么错，以及下一次如何说得更自然。

API 模式下，系统会尝试请求模型返回结构化 JSON，包括 AI 回复、纠错和评分。如果 API 调用失败或返回格式不稳定，系统立即降级到本地规则模式，避免页面崩溃。

## 7. 口语能力量化评价指标

系统采用五维评分：Pronunciation 发音清晰度、Fluency 流利度、Grammar 语法准确度、Expression 表达自然度、Completeness 回答完整度。每个维度分数为 0 到 100，并配有文字解释。

由于第一版没有接入真实发音评测服务，Pronunciation 分数是模拟评分。模拟评分会根据是否使用音频流程和文本长度进行调整，并在页面上明确说明。Fluency 主要考虑回答长度和句子连贯性，Grammar 根据纠错问题数量扣分，Expression 根据表达丰富度进行估计，Completeness 根据回答是否包含足够信息进行估计。虽然该评分不是严格的语言学测评，但足以支撑 MVP 的流程展示和比赛项目分析。

## 8. 本地演示模式与 API 模式设计

双模式设计是本项目的关键。API 模式适合后续优化，可生成更自然、更开放的对话和总结。本地演示模式则保证当前版本不产生额外费用，也不依赖外部服务。系统通过环境变量 `OPENAI_API_KEY` 判断是否进入 API 模式，但即使检测到 Key，API 调用失败后也会自动降级。

本地演示模式并不是空壳，而是完整覆盖主流程。它能根据场景生成预设回复，根据关键词做简单变化，完成规则纠错、五维评分、总结生成和历史保存。因此比赛展示时即使没有网络、没有 API Key、没有语音模型，也可以完成从输入到导出的完整演示。

## 9. 系统测试用例

测试用例一：面试场景正常文本输入。选择 Interview 和中等难度，输入项目经历相关英文，系统应返回面试官追问、纠错反馈、五维评分，并保存 JSON 历史。

测试用例二：点餐场景正常文本输入。选择 Restaurant Ordering，输入 `I would like to order a chicken salad.`，系统应返回服务员角色回复，例如询问饮品或其他需求。

测试用例三：会议场景正常文本输入。选择 Business Meeting，输入关于方案风险或建议的句子，系统应返回会议同事式追问。

测试用例四：语法错误输入。输入 `I am agree we should discuss about this topic`，系统应识别 `I am agree` 和 `discuss about` 的问题，并给出修改句。

测试用例五：空输入。用户不输入内容直接点击提交，系统应显示提示信息，不生成错误记录，不发生崩溃。

测试用例六：上传音频但语音模型不可用。系统应提示当前为演示模式，允许用户继续使用文本输入。

测试用例七：生成并导出课后总结。完成至少一轮对话后点击生成总结，系统应显示 Markdown 总结，提供下载按钮，并在 `outputs/` 中保存总结文件。

## 10. 总结与展望

本项目完成了 AI 英语口语陪练 V1.0 的核心功能。系统能够本地运行，支持场景选择、结构化对话、音频上传与转写降级、AI 角色回复、语法纠错、五维评分、课后总结、Markdown/HTML 导出、JSON 历史保存、历史筛选管理和学习统计。项目结构模块化，便于后续继续优化。

未来可以从三个方向扩展。第一，增强智能反馈能力，使用更稳定的结构化 API 输出，增加更多语法和表达规则。第二，增强语音能力，接入真实发音评测或使用本地声学特征分析，让 Pronunciation 分数更加可信。第三，增强学习管理能力，例如加入用户练习曲线、错题本、每日任务和多场景练习计划。总体来看，本项目已经具备比赛项目第一版交付所需的完整性和可展示性。

## 11. V1.0 完整版扩展说明

在 MVP 的基础上，V1.0 版本进一步补齐了比赛项目中更适合展示的功能分支。首先，系统增加了结构化练习步骤，每个场景都包含四个阶段任务，例如面试场景包含自我介绍、项目经历、问题解决和岗位匹配，会议场景包含表达观点、风险分析、行动安排和达成一致。页面会根据当前对话轮数显示练习完成度、当前任务、建议追问和目标表达，使用户不只是随意聊天，而是沿着结构化口语训练路径逐步完成练习。

其次，系统增加了完整历史管理和学习统计功能。历史记录不再只是保存到 JSON 文件，用户可以在页面中按场景筛选、查看详情、下载单条 JSON 记录，也可以删除不需要的记录。学习统计模块会读取所有历史记录，计算练习次数、平均分、最高分、总结生成数量、场景分布、分数趋势、五维平均分和高频错误。这些统计结果不仅能帮助用户观察进步，也能为比赛项目说明提供更直观的实验数据。

最后，系统增加了报告导出中心。用户可以从当前会话或历史记录中选择已有总结，下载 Markdown 文件，或生成 HTML 学习报告。HTML 报告采用轻量模板，适合浏览器打开、打印或截图。通过这些扩展，项目从“可运行的 MVP”升级为“可演示、可分析、可提交报告的 V1.0 完整版比赛项目系统”。
